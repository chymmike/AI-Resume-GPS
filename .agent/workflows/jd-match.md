# JD 適合度分析 Workflow

當使用者貼入一份 Job Description 並觸發此 workflow 時，執行以下步驟：

## 步驟

### 1. 讀取使用者個人資料

讀取以下兩份檔案，完整理解使用者的背景：

- `data/master_profile.json`（若不存在則使用 `data/master_profile.example.json`）— 技能、經歷摘要、學歷
- `data/experience_notes.md` — 各份工作的詳細描述、量化數據、實際職責

### 2. 解析 JD 關鍵需求

從使用者貼入的 JD 中提取：

- **硬性要求 (Required)**：必備技能、年資、產業經驗等
- **加分項 (Preferred)**：有的話加分但非必要的條件
- **職位核心本質**：這個角色日常到底在做什麼（e.g. B2B 客戶對接、數據分析、專案管理等）

### 3. 逐項對照比對

將 JD 的每項要求與使用者資料逐一比對，產出比對表格：

| JD 要求 | 對應的個人經驗 | 符合程度 |
|---------|--------------|---------|
| 例：SQL & Python | Python (pandas), SQL | ✅ 完全符合 |
| 例：機構客戶經驗 | 無直接經驗 | ❌ 缺乏 |

符合程度分為：
- ✅ 完全符合
- 🟡 部分符合（需包裝）
- ❌ 缺乏

### 4. Risk Signal 掃描

針對個人資料與 JD 的組合，掃描以下 AI HR 常見的危險信號：

| Risk Signal | 檢查項目 | 結果 |
|-------------|----------|------|
| 任期空缺 | 工作時間軸是否有超過 3 個月的空缺？ | ✅ / ⚠️ |
| 職稱層級 | 個人職稱是否與 JD 要求的層級明顯落差（過高/過低）？ | ✅ / ⚠️ |
| 量化數據 | 是否有 bullet 沒有量化數據，可能被判定為「無法驗證」？ | ✅ / ⚠️ |
| 職涯敘事連貫性 | 跳槽脈絡是否清晰？是否有「為何換到這個方向」的疑問？ | ✅ / ⚠️ |
| 關鍵字覆蓋 | Required Skills 是否在現有履歷中完全被提及？ | ✅ / ⚠️ |

### 5. 產出結構化報告

以繁體中文輸出以下結構：

```
## 整體適合度：XX%

### ✅ 優勢（直接命中）
- ...

### ⚠️ 劣勢（缺乏或需大幅包裝）
- ...

### 🚨 Risk Signals（AI HR 可能標記的危險項目）
- [ ] 任期空缺：...
- [ ] 職稱層級：...
- [ ] 量化數據缺失：...
- [ ] 職涯敘事問題：...
- [ ] 關鍵字未覆蓋：...

### 💡 包裝建議
針對劣勢項目和 Risk Signals，具體建議如何在履歷中調整措辭或強調哪些經驗來補強。

### 🎯 結論
一句話總結：值不值得投遞，以及投遞時的策略重點。
```

## 注意事項

- 全程使用**繁體中文**回答
- 不要給高層次的廢話，要給**具體的分析和建議**
- 符合程度要基於實際經驗比對，不要灌水
- Risk Signals 必須逐項說明，不可略過
- 如果使用者想進一步產出客製化履歷，提示他可以用 `/jd-resume` 來生成
- 產出履歷後可以接著用 `/jd-aiproof` 進一步模擬 AI HR 審查
