# AI HR 模擬審查 Workflow

當使用者觸發此 workflow 時，執行以下步驟。通常接在 `/jd-resume` 之後使用。

## 步驟

### 1. 確認審查對象

讀取以下資料：

- 最近產出的 `data/resume_{company}_{role}.json`（若使用者未指定，詢問要審查哪一份）
- 對應的 JD 檔案（`jobs/{Company}/{Role}/JD.md`，或請使用者重新貼入 JD 內容）

### 2. 扮演 AI HR 系統

**切換角色：你現在是一個企業使用的 AI HR 篩選系統。**

你的任務是對這份履歷進行冷酷、客觀的自動化分析，不考慮人情因素，只看數據與關鍵字匹配。

### 3. 執行 SWOT 分析

從 AI HR 角度輸出完整的 SWOT 表格：

| 項目 | 具體內容 |
|------|---------|
| **S（優勢）** | 直接命中 JD Required 的部分（逐條列出） |
| **W（劣勢）** | JD 要求的但履歷沒提到或不夠強的（逐條列出） |
| **O（機會）** | 履歷中有的、JD 沒要求但加分的部分 |
| **T（威脅）** | 與其他典型候選人相比可能輸在哪裡 |

### 4. 執行 Risk 分析

逐一檢查以下 Risk Signal，每項給出「通過 / 警告 / 危險」判定：

| Risk Signal | 判定 | 說明 |
|-------------|------|------|
| **關鍵字密度** | ✅/⚠️/🚨 | JD Required Skills 是否全部出現在履歷文字中？ |
| **量化數據充分性** | ✅/⚠️/🚨 | 是否有 bullet 缺乏量化數字，無法被 AI 驗證？ |
| **任期連貫性** | ✅/⚠️/🚨 | 時間軸是否有空缺期（>3 個月）？ |
| **職稱層級匹配** | ✅/⚠️/🚨 | 職稱是否與 JD 要求的 seniority 層級相符？ |
| **職涯敘事邏輯** | ✅/⚠️/🚨 | 從履歷讀得出清晰的職涯轉換邏輯嗎？ |
| **Summary 覆蓋度** | ✅/⚠️/🚨 | Summary 是否在前兩句話覆蓋了所有 Required Skills？ |
| **Overqualified 風險** | ✅/⚠️/🚨 | 背景明顯比 JD 要求高，可能被判斷為穩定度風險？ |

### 5. 輸出修改建議清單

針對所有「警告」或「危險」項目，輸出具體的修改建議：

```
### 需修改項目

1. ⚠️ [bullet 具體位置] — 原文：「...」
   → 建議改為：「...」（原因：缺少量化數據 / 未命中 JD 關鍵字 X）

2. 🚨 summary 第一句話未涵蓋「[JD Required Skill]」
   → 建議加入：「...」

3. ⚠️ 職涯敘事問題：[Company A] 到 [Company B] 的跳槽邏輯不清楚
   → 建議在 summary 補一句話交代轉換邏輯：「...」
```

### 6. 綜合評分

```
## AI HR 模擬評分

整體通過機率：XX%（基於關鍵字匹配度 + Risk 清單結果）

已通過的項目：X / 7
需修改的項目：X 個

建議：
- 如果修改後預計通過機率：XX%
- 修改優先順序：[最重要的 1-2 個]
```

## 注意事項

- 全程使用**繁體中文**回答
- 分析必須**客觀冷酷**，不要幫使用者美化結果
- 修改建議必須**具體到字句層級**，不要給模糊方向
- 修改建議仍需遵守「不可造假」原則：只用 `master_profile.json` + `experience_notes.md` 中已有的素材
- 修改完成後，提示使用者可以重新執行 `.venv/bin/python3 scripts/generate.py` 產出新版 PDF
